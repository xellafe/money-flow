#### Flow 1 — Background Update Check → Banner (happy path ✓, error path ✗)
`autoUpdater.checkForUpdates()` (3s startup) → `update-available` event → `webContents.send` → preload listener → `useUpdateStatus` → auto `startDownload()` → progress events → `status='ready'` → `UpdateBanner` renders in `App.jsx`

#### Flow 2 — Manual Check in Settings ✓
User clicks "Controlla aggiornamenti" → `status='checking'` (spinner shown) → IPC invoke → `autoUpdater.checkForUpdates()` → error forwarded via `webContents.send('updater:error')` → all 3 Settings states reachable

### ⚠ Integration Issues (Tech Debt)

#### 🔴 Issue 1 — Background/Download Errors Never Reach Renderer [Medium]
**Affects:** UPD-09 (error state only via manual check)  
**Location:** `electron/main.cjs` lines 72–76

The `autoUpdater.on('error', ...)` handler only calls `log.error()`. It does **not** call `mainWindow.webContents.send('updater:error', ...)`. This means:
- Background startup check failures → renderer stays on `status='idle'` forever
- Mid-download failures → renderer stays on `status='downloading'` forever  

The `updater:error` push only fires from the `updater:check-for-updates` IPC handler (line 345) — manual checks only.

**Fix:** Add `mainWindow.webContents.send('updater:error', err.message)` inside `autoUpdater.on('error', ...)`, OR document as intentional silent-fail design.

#### 🟡 Issue 2 — `startDownload()` Failure Silently Swallowed [Medium]
**Affects:** UPD-04, UPD-05 (banner never appears if download fails)  
**Location:** `electron/main.cjs:351–359` vs `src/hooks/useUpdateStatus.js:31–34`

The `updater:start-download` IPC handler catches all errors and **resolves** with `{ success: false }` instead of rejecting. The hook calls `.catch()` which never fires. If `downloadUpdate()` throws, `status` stays `'available'`, no error shown, banner never appears.

**Fix A:** Throw inside handler on error → `.catch()` in hook fires as intended.  
**Fix B:** Inspect return value: `const res = await startDownload(); if (!res?.success) { setStatus('error'); setError(res?.error); }`

#### 🟡 Issue 3 — `publish` Config Under `nsis` Only, Not Top-Level [Low]
**Affects:** UPD-01 (partial — NSIS only)  
**Location:** `package.json` `build.nsis.publish`

Portable builds have no `publish` config → no `app-update.yml` → `checkForUpdates()` throws at runtime on portable. Likely intentional (portable cannot self-replace), but the production guard (`!isDev`) does not check for portable build, so the 3s startup check fires and logs silently.

**Fix (if desired):** Add runtime portable detection guard, or add top-level `build.publish` if NSIS+portable both need update support.
